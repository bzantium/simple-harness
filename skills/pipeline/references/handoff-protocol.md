# Handoff Protocol

All agent communication happens through files in the `.simple/` directory. This document defines the schema and lifecycle for each file.

## Directory Structure

```
.simple/
├── spec.md              # Planner output (contract)
├── changes.md           # Generator output (what was done)
├── evaluation.md        # Evaluator output (verdict)
├── checkpoint           # Git HEAD hash before Generator started (for rollback on FAIL)
├── notepad.md           # PreCompact hook output (compaction resilience)
├── gotchas.md           # Stop hook output (cross-session learning)
├── changes-{date}.md    # Archived changes from completed pipelines
└── evaluation-{date}.md # Archived evaluations from completed pipelines
```

## File Lifecycle

### During a pipeline run:

```
User prompt
  │
  ▼
Planner writes spec.md
  │
  ▼
User approves spec
  │
  ▼
Generator reads spec.md, writes changes.md
  │
  ▼
Evaluator reads spec.md + changes.md, writes evaluation.md
  │
  ├─ PASS → archive changes.md & evaluation.md → done
  │
  └─ FAIL → Generator reads evaluation.md, updates changes.md
            → Evaluator re-evaluates (max 2 retries)
```

### Concurrency

Only one pipeline runs at a time. Generator and Evaluator are never dispatched in parallel.

### Persistence

- `spec.md` persists across pipelines (kept after PASS for reference)
- `gotchas.md` accumulates across sessions (append-only)
- `notepad.md` is transient (cleared after successful pipeline)
- `changes.md` and `evaluation.md` are archived with date suffix after PASS

## spec.md Schema

```markdown
# Spec: {title}

## Goal
{What and why -- one paragraph}

## Success Criteria
- [ ] {Testable, verifiable criterion}

## Files to Change
- {path} -- {what and why}

## Approach
{How to implement -- 2-5 paragraphs}

## Risks
- {Risk and mitigation}

## Open Questions
- {Questions for user, or empty}
```

**Rules:**
- Max 200 lines
- Success criteria must be binary (pass/fail), not subjective
- No implementation code in the spec

## changes.md Schema

```markdown
# Changes

## Files Modified
- {path} -- {what changed}

## Files Created
- {path} -- {purpose}

## Tests
- {command}: {PASS/FAIL with details}

## Notes
- {Deviations, blockers, edge cases}
```

## evaluation.md Schema

```markdown
# Evaluation

## Verdict: {PASS | FAIL}

## Criteria Results
- [x] {criterion} -- PASS: {evidence}
- [ ] {criterion} -- FAIL: {what's wrong}

## Test Results
- {command}: {result}

## Regressions
- {None | list}

## Feedback for Generator
{Specific, actionable fix instructions with file:line references}
```

## gotchas.md Schema

Append-only format:

```markdown
## {YYYY-MM-DD}
- [HIGH] {category}: {description} -> {suggested fix}
- [MEDIUM] {category}: {description} -> {suggested fix}
```

Categories: `user-correction`, `repeated-instruction`, `pipeline-failure`, `classification-error`, `compaction-gap`

## notepad.md Schema

Auto-generated by the PreCompact hook. Contains snapshots of active spec, latest evaluation, and known gotchas. Not manually edited.
